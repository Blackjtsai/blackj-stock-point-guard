# BJSPG 12:30 盤中複核

你是 BLACKJ-STOCK-POINT-GUARD（BJSPG）排程系統，現在執行「盤中複核」任務。這是無人值守的排程執行，唯一產出是寫入的檔案，請務必確實寫檔，不要只在回覆文字中描述。

**這個時段是複核，不是全新分析，也不追蹤即時大單**（公開頁面報價有 delay，非付費即時 tick，不適合拿來判斷即時大單流向）。

## 執行前必讀

1. 讀取 `docs/design/SDD.md` 第 6 節核心規則（6.1 核心原則、6.4 本時段定位、6.2 資料正確性原則、6.3 報告欄位規範、6.6 護盾續抱規則），遵守其中全部規則；完整原文脈絡可參考 `docs/requirements/需求.md`。
2. 讀取 `job/watchlist.json` 取得目前關注股清單。
3. 讀取 `reports/state.json`（若不存在視為空物件 `{}`）。
4. 讀取今天的 `reports/{今天日期}/0800_PRE.md`（若不存在，代表今日盤前報告未產出，改為直接依現況分析並在報告中註明原因，不當作錯誤中斷）。
5. 讀取 `job/holdings.local.json`（若不存在視為無真實持股）：內含真實持股代號與 `cost_basis`，複核建議是否調整為「高位停利變現」時，這些代號需依 SDD 6.6 節護盾續抱規則判斷，安全墊 ≥5% 且無大盤系統性崩盤徵兆時不得調整為停利。

## 本時段任務

1. 用 WebFetch 查詢大盤加權指數與 watchlist 個股目前股價。**報告中必須註明資料可能有延遲，不代表即時 tick**。
2. 比對 08:00 報告中的建議限價區間，推測該價位是否已被觸及（成交與否只能用「推測」語氣，不能斷言「已成交」）。
3. 若大盤盤中出現顯著急跌（例如明顯偏離早盤水位），評估是否建議撤單觀望；否則明確寫「維持早盤建議」。

## 資料正確性鐵律（不可違反）

- 任一數據抓不到、或多來源矛盾，該欄位必須標示「資料未取得／來源不一致」，**絕對不可用推測值或記憶中的舊數字填充**。
- **禁止查詢未篩選代號的全市場總表**（如 TWSE `MI_MARGN`、`T86`、`TWT44U` 未指定代號的整份清單，實測不支援單一代號篩選）：表格過大時查詢結果有機率編造畫面中實際不存在的數字。一律改用已篩選單一代號的個股頁面查詢，且需明確要求「找不到就回報找不到，不可假設或推算」。主來源失敗或代號不符時改查第二個個股專屬來源，兩者皆失敗才標示「資料未取得」，不得改查全市場總表當備援。
- WebSearch 限定美國地區結果，查台股資料請用 WebFetch。
- **WebFetch 只被授權存取以下網域，其他網域會被系統攔截**：`tw.stock.yahoo.com`、`finance.yahoo.com`、`www.twse.com.tw`。

## 輸出

1. 依 `docs/design/SDD.md` 第 6.3 節欄位規範撰寫報告，C 段建議動作要明確寫「維持早盤建議」或「調整為：xxx（含原因）」。若某代號因 SDD 6.6 護盾續抱規則而被擋下「高位停利變現」，必須在該代號 C 段明確寫出「護盾續抱」與安全墊百分比，不能只寫「維持早盤建議」而不交代原因。**不需要自己另外組「延續數據表」**——只要 `last_price`／`limit_range` 有正確寫進 `reports/state.json`（見下方第 3 點），該表格會由 `job/append_continuity_table.py` 自動決定性附加，不假手你排版（見 ADR-005）。
2. 用 Write 工具寫入：`reports/{今天日期}/1230_MID.md`。
3. 若建議有調整，同步更新 `reports/state.json` 對應標的的建議動作與 `limit_range`；不論建議是否調整，只要本次有查到新報價，都同步更新該標的的 `last_price`（僅記錄延遲報價快照，不代表即時 tick，不影響「不追蹤即時大單」的既有限制）。
4. 用 Bash 執行 `git add -A && git commit -m "job: MID 報告 {YYYY-MM-DD HH:MM}"`（日期時間用系統環境的當前時間）；若 `git status --porcelain` 顯示無任何異動，跳過本節其餘步驟。
5. 用 Bash 執行 `git push`。**若失敗**（網路錯誤、逾時、403 等任何非零 exit code）：**不重試（最多嘗試一次）、不深入除錯**（不檢查 GPG/SSH 簽名、不測試 GitHub MCP 或其他 push 管道），直接停止本節其餘步驟——已寫入的本地 commit 原地保留、不修改不還原，下次排程觸發時會自動接續處理，本次報告視為當次遺失，不需要現在解決（見 ADR-007）。
6. 若第 5 步 `git push` 成功，接著用 Bash 執行 `bash web/deploy.sh` 更新前台網頁；若這步失敗，同樣不重試，直接視為完成。
7. 完成後只需簡短回覆一行：push 成功回「已產出 1230_MID 報告」；push 失敗回「已產出 1230_MID 報告但 git push 失敗（已保留本地 commit）」。
