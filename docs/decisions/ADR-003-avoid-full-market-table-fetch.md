# ADR-003：禁止 WebFetch 查詢未篩選代號的全市場總表

> 狀態：已採納
> 日期：2026-07-04
> 決策者：黑暗傑客（Eason Tsai）／Claude Code

## 背景

在手動執行一次 12:30 / 21:30 排程邏輯驗證時（Layer 2），查詢大聯大(3702) 的融資融券資料，先用 WebFetch 打 TWSE 全市場總表 `MI_MARGN`（未篩選代號，包含全部上市股票）。因表格過長被截斷，WebFetch 的摘要模型**回傳了一組看似合理、格式正確、但實際上該股票根本沒有出現在畫面內容中的數字**（融資 +445／融券 +5），與後續改查 Yahoo 個股頁面取得的正確數字（融資 -140／融券 +404，與同日 08:30 報告一致）明顯矛盾。

換一個更明確的提示詞（要求「找不到就明確說找不到，不要用其他列的數字替代」）後，同一個 TWSE 全市場總表才如實回報「找不到 3702」。這證實：**問題不在資料本身取得不到，而在於「全市場大表格 + 預設寬鬆提示詞」的組合，讓摘要模型在找不到目標列時傾向編造一個格式正確的答案，而不是誠實說找不到。**

這直接牴觸專案的「資料正確性鐵律」：抓不到資料時絕對禁止捏造，但目前 prompt 設計本身沒有針對「全市場大表格容易被摘要編造」這個特定風險做防範，等於留了一個會產生看似合理但實為捏造數字的漏洞，且不易被使用者察覺（因為格式和其他真實數據一模一樣）。

## 選項

| 選項 | 優點 | 缺點 |
|---|---|---|
| A. 不改，靠「提示詞明確要求找不到要如實說」逐次防範 | 不用改架構 | 排程是無人值守執行，沒有人會在當下把提示詞越改越保守；且已證實預設提示詞下會出錯 |
| B. 明訂規則：一律改用已篩選單一代號的個股頁面（Yahoo 奇摩股市個股頁），禁止查詢未篩選代號的全市場總表 | 個股頁面內容小、不會被截斷，摘要模型沒有「編造未顯示內容」的空間；已實測數字與可信來源一致 | TWSE 官方全市場總表理論上是「當日正式公告」的第一手來源，改用 Yahoo 個股頁面等於改變主要資料來源 |

## 決策

選擇 **選項 B**。

理由：`docs/design/SDD.md` 第 6.2 節「資料正確性原則」的精神是「抓不到就明確標示，不可捏造」，但這次發現的問題比抓不到更隱蔽——是「格式正確但內容捏造」，使用者難以從報告表面察覺異常。從根源上避免摘要模型有機會接觸到會被截斷的大表格，比依賴提示詞措辭更可靠。Yahoo 個股頁面雖非官方原始來源，但本專案原本就已在多處使用 Yahoo 作為輔助/交叉比對來源，且本次驗證顯示其個股融資融券數字與 TWSE 官方數字對得上（同一份 07-03 資料）。

後果：
- `docs/design/SDD.md` 第 6.2 節新增規則（v0.2 → v0.3），`job/blueprint.md`、`job/prompts/{PRE,MID,POST}.md` 的「資料正確性鐵律」段落同步新增此條。
- **代價**：三大法人／融資融券資料改以個股頁面為主要來源，若之後 Yahoo 個股頁面改版或下架，需重新評估替代來源；TWSE 官方全市場總表仍可作為官方佐證，但查詢時必須篩選單一代號（若 TWSE 有提供代號篩選參數）或直接放棄使用未篩選版本。
